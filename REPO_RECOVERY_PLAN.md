## Needs review â€” decisions required

### 1. Root-level Java/Gradle scaffold project
**Files:** `app/`, `build-logic/`, `list/`, `utilities/`, `settings.gradle.kts`, `gradlew`, `gradlew.bat`, `gradle/`, `gradle.properties`, `build.gradle.kts`

**Finding:** These match the output of `gradle init --type java-application` exactly. Package names (`org.example.app`, `org.example.list`, `org.example.utilities`) and classes (`App.java`, `LinkedList.java`, `JoinUtils.java`) are boilerplate. There is no reference to HappiTime in any of these files. The `README.md` makes no mention of Java or Gradle.

**Risk:** Unknown. It's possible these were committed accidentally from a side project or tutorial that was open in IntelliJ at the same time as HappiTime. It's also possible they anchor the IntelliJ `.idea/` configuration.

**Recommended action:** If no one on the team recognizes this Java project, remove it:
```bash
git rm -r app/ build-logic/ list/ utilities/ settings.gradle.kts gradlew gradlew.bat gradle/ gradle.properties build.gradle.kts
```
Then commit. The `.idea/` files that reference these modules (`.idea/modules.xml`, `HappiTime.iml`, etc.) can be trimmed at the same time.

### 2. Oracle JDK committed to history
**Commit:** `cfdd62e8` added it; `1b23f3b` deleted it with "Delete oracle JDK"

**Finding:** A full Oracle JDK 26 (binaries) was committed to the repository. Even though it was deleted, the binary content remains in every clone's `.git/objects`. This adds hundreds of MB to clone size.

**Risk:** History rewrite is destructive and requires coordination across all forks/clones.

**Recommended action (when you have team coordination):**
```bash
# Requires git-filter-repo (brew install git-filter-repo)
git filter-repo --path oracleJdk-26.jdk --invert-paths
git push --force-with-lease origin master
```
All contributors must re-clone or run `git fetch --all && git reset --hard origin/master` after this.
